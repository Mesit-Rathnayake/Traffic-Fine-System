# 🚦 Traffic Fine System Mobile App - Architecture

## 📌 Summary

This Flutter application is a lightweight, multi-platform client for managing and paying traffic fines. It focuses on a simple, form-driven UI (login, signup, fine lookup and payment) with mock payment processing. There is no backend integration or external state-management library in the current codebase most logic is local to widgets.

## 🧩 High-level Components

- 🎨 **UI Layer**: Implemented with Flutter widgets. The main entry and theme live in [lib/main.dart](lib/main.dart). The primary feature screen is the payment/home screen at [lib/screens/home_page.dart](lib/screens/home_page.dart).

- 🧭 **Routing / Navigation**: Uses `MaterialApp` named routes defined in `main.dart` (routes: `/signup`, `/home`, `/payment-home`) and `Navigator` calls (`pushNamed`, `pushReplacementNamed`).

- 🧠 **State Management**: Local widget state via `StatefulWidget`, `TextEditingController`, and `Form`/`GlobalKey<FormState>`. No global state store (Provider, Riverpod, Bloc, etc.) is present.

- 🗄️ **Data Layer**: No network or persistence layer exists. Payment processing and form submission are mocked (see `_submit()` in `HomePage` which uses `Future.delayed`).

- 🔌 **Platform Integration**: Standard Flutter platform folders are present: Android, iOS, Web, Windows, macOS, Linux. Android application id is set in [android/app/build.gradle.kts](android/app/build.gradle.kts). iOS bundle id is in Xcode project settings referenced from [ios/Runner/Info.plist](ios/Runner/Info.plist).

## 📝 Detailed Notes

- 🎨 **UI & Theming**: `main.dart` defines a seed `ColorScheme`, custom `InputDecorationTheme` and `ElevatedButtonTheme`. The app uses Material design components and is responsive via `ConstrainedBox` and `MediaQuery` checks in `HomePage`.

- 🧾 **Forms & Validation**: `HomePage` and auth screens use `Form` with `TextFormField` validators and `GlobalKey<FormState>` patterns. Payment form contains client-side formatting helpers (card formatting, expiry formatting) and validation.

- 💳 **Payment Flow**: Payments are currently simulated inside `_submit()` — a delay + SnackBar. There is no integration with payment SDKs (Stripe, PayPal) or PCI-compliant handling.

- 🌐 **Networking & Persistence**: There are no HTTP clients, database, or local persistence implementations. No secure storage or token handling is present.

- 🧪 **Tests**: Project contains default Flutter widget test only. No unit tests for business logic or integration tests for flows.

## 🖥️📱 Supported Platforms

The repository contains platform folders for Android, iOS, Web, macOS, Windows, and Linux. Out-of-the-box the project can be built to those platforms using standard Flutter tooling.
